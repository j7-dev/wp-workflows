# Web Backend 句型分析方針

## 1. 可測試性前提

系統具有兩層：
- **API Layer**（HTTP endpoint）— 提供 Operation Invocation 的調用介面
- **持久化 Layer**（DB / Repository）— 提供 States Prepare 的直接注入 + States Verify 的查詢介面

這兩層的存在使以下 6 種 Handler 成立。

---

## 2. 後設決策樹（4 抽象角色）

```
Gherkin Step
    │
    ├─ Given? ─── 句子在描述什麼？
    │   │
    │   ├─【狀態】（靜態、存在式）
    │   │   「系統中有…」「…的…為…」「…包含…」「…存在…」
    │   │   → States Prepare
    │   │
    │   └─【已完成的動作】（動態、過去式/完成式）
    │       「已訂閱」「已建立」「已註冊」「已完成」「已添加」
    │       → Operation Invocation（作為前置操作，預期成功）
    │
    ├─ When? ─── Operation Invocation（作為測試目標，可能成功或失敗）
    │   │
    │   ├─ 修改操作（Mutation）
    │   │   動詞改變系統狀態：「更新」「建立」「刪除」「提交」「添加」「移除」
    │   │
    │   └─ 讀取操作（Query）
    │       動詞不改變狀態：「查詢」「取得」「列出」「檢視」「搜尋」
    │
    └─ Then? ─── 驗證的資料從哪來？
        │
        ├─ 從【操作的回傳值】取得（When 已拿到的資料）
        │   → Operation Result Verifier
        │   │
        │   ├─ 只看成敗（不提及任何資料欄位）
        │   │   「操作成功」「操作失敗」「錯誤訊息應為 "xxx"」
        │   │
        │   └─ 看具體資料欄位
        │       「查詢結果應包含進度 80%」「結果中應有 3 筆」
        │
        └─ 需要【額外查詢系統內部狀態】才能取得
            → States Verify
            「…的…應為…」「應包含以下…」（驗證儲存層，非回傳值）
```

### 判斷口訣

| 分岔 | 口訣 |
|------|------|
| Given：States Prepare vs Operation Invocation | 「**有**什麼」→ States Prepare；「**做過**什麼」→ Operation Invocation |
| When：Mutation vs Query | 執行後系統**變了** → Mutation；系統**沒變** → Query |
| Then：Result Verifier vs States Verify | 用 When **已拿到**的資料驗 → Result Verifier；需要**再去查** → States Verify |
| Result Verifier：成敗 vs 資料 | 只說「成功/失敗」→ 成敗；提到**具體欄位值** → 資料 |

## 3. 抽象 → Web Backend 映射表

| 抽象角色 | 分支 | Web Backend Handler | 實現方式 |
|---------|------|--------------------| --------|
| States Prepare | — | `aggregate-given` | Repository.save() 直接寫 DB |
| Operation Invocation | Given + Mutation | `command` (Given) | HTTP POST/PUT，預期成功 |
| Operation Invocation | When + Mutation | `command` (When) | HTTP POST/PUT/DELETE |
| Operation Invocation | When + Query | `query` | HTTP GET |
| Operation Result Verifier | 成敗 | `success-failure` | HTTP status code |
| Operation Result Verifier | 回傳資料 | `readmodel-then` | response.json() 欄位 |
| States Verify | — | `aggregate-then` | Repository 查 DB 驗屬性 |

## 4. Handler 間的區別表

### States Prepare vs Operation Invocation (Given)

| | States Prepare (aggregate-given) | Operation Invocation (command as Given) |
|---|---|---|
| 目的 | 直接建立 DB 資料（繞過 API） | 透過 API 執行命令 |
| 層級 | Repository 層 | HTTP API 層 |
| 語態 | 現在式/存在式（「有」「為」） | 過去式/完成式（「已訂閱」「已建立」） |

### Operation Invocation：Given vs When

| | Given | When |
|---|---|---|
| 角色 | 建立前置條件 | 測試目標操作 |
| 失敗處理 | 不預期失敗 | 可能成功或失敗 |

### Mutation vs Query

| | Mutation (command) | Query (query) |
|---|---|---|
| HTTP | POST/PUT/PATCH/DELETE | GET |
| 系統狀態 | 修改 | 不修改 |
| Then 搭配 | success-failure / aggregate-then | readmodel-then |

### 三種 Then 互比

| | success-failure | readmodel-then | aggregate-then |
|---|---|---|---|
| 抽象角色 | Result Verifier (成敗) | Result Verifier (資料) | States Verify |
| 資料來源 | HTTP status / exception | response.json() | Repository 查詢 |
| 前提操作 | Command | Query | Command |

## 5. 使用方式

- **bdd-analysis（分析時）**：用 §2 的決策樹分類每個 Gherkin 句子，用 §3 對應到具體 handler type
- **auto.red（實作時）**：用 §3 找到具體 handler → 載入 `handlers/{type}.md` + `variants/{variant}.md`

完整 handler 定義見 `handlers/*.md`（含 Trigger 辨識、實作流程、BDD 模式、共用規則）。
語言變體見 `variants/*.md`（含程式碼模式、框架 API、DI 機制）。
