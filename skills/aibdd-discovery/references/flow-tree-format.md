# Flow Tree 格式定義

純文字 tree 格式，直接在對話中呈現。**不寫入任何 `.mmd` / `.activity` / `.feature` 檔案。**

## 格式元素

| 元素 | 說明 |
|------|------|
| `N. @<Actor> <動作>` | 流程步驟，標記執行者 |
| `<- 原文「...」` | 追溯到使用者原文（proof of non-fabrication） |
| `[?]` | 使用者未提及、需要補充 |
| `<- NEW` | 迭代中新增的步驟（v2+） |
| `<- CONFIRMED` | 原本是 `[?]`，使用者已確認 |
| `[<子流程名>]` | 獨立子流程（不同 Actor 或不同生命週期） |
| `待釐清：` | 所有 `[?]` 的彙整 |

## 範例

```
<流程名>（v<N> 草稿）

[?] <缺失的前置條件或未知步驟>
 |
 1. @<Actor> <動作描述>
 |  <- 原文「<使用者原文片段>」
 |
 2. @<Actor> <動作描述>
 |  <- 原文「<使用者原文片段>」
 |  [?] <此步驟的未解疑問>
 |
 3. @[?] <動作描述，但不確定誰執行>
    [?] <觸發者未明>

待釐清：
 - [?] <問題 1>
 - [?] <問題 2>
```

## 多流程情境

若從 idea 識別出多個獨立流程（不同 Actor 群組、不同生命週期），以 `[<流程名>]` 分區：

```
CRM 系統（v1 草稿）

[管理員設定流程]
 1. @管理員 建立 Journey 範本（含 Stage 定義）
 |
 2. @管理員 建立 SOP
 |
 3. @管理員 綁定 SOP 到 Stage

[客服日常流程]
 1. @客服人員 查看學員名單
 |  <- 原文「有一大堆的學員名單」
 |
 2. @客服人員 查看學員的 Journey 進度
 |  <- 原文「每個名單都可以有狀態轉變，並擁有自己的 journey」
```

## 迭代循環

```
展示 Flow Tree（vN）
  ↓
詢問：「這個流程大方向對嗎？有沒有要新增、刪除、或調整的步驟？確認請說 confirm。」
  ↓
使用者回覆 →
  ├─ 提出修改 → 更新 Flow Tree 為 v(N+1)，標記差異（<- NEW / <- CONFIRMED），重新展示
  ├─ 提出新的 [?] 問題 → 加入待釐清清單，重新展示
  └─ 說 "confirm" → 結束流程梳理，進入 Step 3
```
