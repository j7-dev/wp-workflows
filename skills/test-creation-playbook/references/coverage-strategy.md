# 測試覆蓋策略：E2E / IT / UT 分工

三層測試不是「做越多越好」，而是**各司其職**。混淆層級會產生大量重複測試與維護負擔。

---

## 層級關注點對照表

| 層級 | 關注點 | 為什麼 | 成本 |
|---|---|---|---|
| 單元測試（UT） | **邊緣案例 + 演算法正確性** | 執行快、隔離性強，可用最小成本覆蓋最多破壞性情境 | 低 |
| 整合測試（IT） | **邊緣案例 + 元件互動** | Hook/Filter 交互、REST API 驗證、資料庫約束、Service 異常、權限邊界——這些單元測試難以覆蓋 | 中 |
| E2E 測試 | **核心業務流程** | 驗證「系統裝起來是通的」，不追求邊緣覆蓋（成本太高） | 高 |

---

## E2E 測試：只測核心路徑

### 原則

- **只覆蓋使用者最關鍵的操作路徑**（購買、註冊、登入等）
- **不展開邊緣案例矩陣**
- 每個核心 flow 一個 happy path，**至多**補一個典型失敗路徑
- E2E 跑得慢、維護成本高、flaky 機率高——保持少而精

### 好的 E2E 範例

1. 新使用者完成註冊 → 登入 → 購買商品 → 收到訂單確認信
2. 管理員登入 → 建立課程 → 指派學員 → 學員看到課程

### 不該寫成 E2E 的情境

- 測試 email 格式驗證（這是 UT 工作）
- 測試權限矩陣的 30 種組合（這是 IT 工作）
- 測試小數點金額計算（這是 UT 工作）

---

## 整合測試：邊緣案例的主戰場

### 原則

- **專注邊緣案例**：Hook/Filter 邏輯、REST API 參數邊界、資料庫約束、Service 層異常處理、權限邊界
- 使用「**功能 × 角色 × 狀態 × 邊界值**」矩陣展開
- 真實 DB 操作（不 mock DB），但可以 mock 外部 service

### 典型覆蓋項目

| 類別 | 覆蓋範例 |
|---|---|
| REST API | 無效參數、缺欄位、型別錯誤、超長輸入、未授權、權限不足 |
| Service 層 | 業務例外、邊界值、並發衝突、交易回滾 |
| Repository | DB 約束違反、唯一索引衝突、外鍵錯誤 |
| Hook/Filter | 註冊順序、priority、回傳值被其他 filter 修改 |

## 單元測試：快速反覆

### 原則

- **專注邊緣案例 + 演算法**：負數、null、邊界值、異常輸入、型別錯誤
- 完全隔離，不碰 DB、不碰檔案系統、不碰網路
- 跑得越快越好（目標 < 1 秒可跑完整個套件）
- 覆蓋率追求高（80%+），但**不以覆蓋率為唯一指標**

### 適合 UT 的情境

- 純函式（validator、formatter、calculator）
- 演算法（sorting、filtering、aggregation）
- 狀態機轉換邏輯
- Reducer / Selector

### 不該寫成 UT 的情境

- 需要真實 DB 的查詢邏輯（改寫 IT）
- 需要真實 HTTP request 的整合（改寫 IT）
- 測試 UI rendering（改寫 Component Test）

## 決策流程圖

```
新功能需求
    │
    ▼
有使用者可見的核心流程？─── YES ──▶ 寫 1-2 個 E2E（只測 happy path）
    │
    NO
    │
    ▼
涉及 DB / API / Hook / 外部依賴？─── YES ──▶ 寫 IT（邊緣案例矩陣展開）
    │
    NO
    │
    ▼
是純邏輯 / 演算法 / validator？─── YES ──▶ 寫 UT（邊緣案例）
    │
    NO
    │
    ▼
重新思考：這個功能真的需要測試嗎？（可能是純樣式調整）
```

## 比例建議

典型 WordPress Plugin 專案的測試數量比例：

- UT: 60-70%
- IT: 25-35%
- E2E: 5-10%

E2E 太多代表「把 IT 該做的事搬到 E2E」，會拖慢 CI、增加 flaky。
UT 太少代表「沒有在最快最便宜的層級攔截問題」。
