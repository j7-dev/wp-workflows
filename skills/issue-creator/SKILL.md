# Issue Creator — 需求展開與結構化生成器

從一句 idea 到完整的 GitHub Issue 需求文件。

核心價值：**用戶只需要說出「我想做什麼」，你負責把它變成可執行的需求規格。**

---

## 工作流程

```
Step 1: 理解用戶的 idea
Step 2: 掃描 specs/ 建立專案上下文
Step 3: 發想延伸 — 邊界情境、依賴、影響
Step 4: 生成結構化需求 markdown
Step 5: 直接輸出到終端機
```

---

## Step 1: 理解用戶的 idea

用戶的輸入可能是：
- 一句話：「我想加一個課程評分功能」
- 一段描述：包含部分背景和期望行為
- 一個 bug 轉需求：「現在 XX 行為不對，應該改成 YY」

無論輸入多粗略，**不要回問太多問題**。先根據專案上下文做出合理假設，把完整需求生出來，讓用戶在看到成品後修改，比來回問答更高效。

如果需求有明顯的歧義（例如同一句話可以解讀成兩種完全不同的功能），才簡短確認。

---

## Step 2: 掃描 specs/ 建立專案上下文

讀取 `./specs` 目錄結構，建立對現有系統的理解：

### 必讀
- `specs/features/` — 所有已定義的 Gherkin feature files，理解系統現有功能邊界
- `specs/activities/` — 業務流程定義，理解功能在流程中的位置
- `specs/open-issue/` — 進行中的需求，避免重複或衝突

### 按需讀取
- `specs/api/api.yml` — 現有 API endpoints，判斷是否需要新增 API
- `specs/entity/erm.dbml` — 資料模型，判斷是否需要新增欄位或表
- `specs/specs/actors/` — 角色定義，確認需求涉及哪些角色
- `specs/specs/ui/` — UI 頁面規格，判斷影響哪些頁面

### 上下文建立的目的

1. **找到相關的現有功能**：新需求可能是現有功能的延伸、修改或衝突
2. **識別影響範圍**：哪些現有 feature/activity 會被影響
3. **發現依賴關係**：新需求是否依賴某個尚未實作的功能
4. **避免重複**：open-issue 中是否已有類似需求

---

## Step 3: 發想延伸

這是 skill 的核心價值。用戶想的是「happy path」，你要替他想到所有他沒想到的。

### 發想維度

#### 3.1 邊界情境
- 空值/零值/極端值怎麼處理？
- 並發操作會怎樣？（兩人同時編輯）
- 權限不足時的行為？
- 資料不存在或已被刪除時？
- 超過限制（數量、大小、長度）時？

#### 3.2 角色視角
參考 `specs/specs/actors/` 中的角色定義，思考：
- 管理員看到什麼？能做什麼？
- 學員看到什麼？能做什麼？
- 講師看到什麼？能做什麼？
- 未登入訪客呢？

#### 3.3 與現有功能的交互
- 這個功能會影響現有的哪些功能？
- 現有的哪些功能需要配合調整？
- 有沒有共用的 UI 元件或 API endpoint 需要修改？

#### 3.4 技術考量
- 前端需要新增 API 呼叫嗎？
- 後端需要新增資料表或欄位嗎？
- 是否涉及第三方服務整合？
- 效能影響？（大量資料查詢、檔案上傳）

#### 3.5 向下相容
- 舊資料如何處理？
- 現有用戶的使用流程會被打斷嗎？
- 是否需要資料遷移？

---

## Step 4: 生成結構化需求

### 輸出格式

使用以下模板，基於 `.github/ISSUE_TEMPLATE/需求模板.md` 但做了擴展。
**每個段落都要有實質內容**，不要留佔位符或空白段落。

```markdown
---
name: 需求模板
about: 需求創建模板
title: "[需求] {簡短標題}"
labels: feature
assignees: j7-dev
---

## 背景資訊

{描述目前系統的相關現狀。包含：}
{- 現有功能是怎麼運作的}
{- 為什麼需要這個新功能/改動}
{- 相關的現有 specs 引用（列出路徑）}

## 任務目標

{條列式說明要做什麼，每一條要具體明確。}
{用「動詞 + 對象 + 條件」的格式，例如：}
{- 新增「XX」按鈕，點擊後開啟 YY 面板}
{- 當 ZZ 條件成立時，自動觸發 AA 動作}

## 情境案例

{用具體的情境描述預期行為，讓讀者能「看到」功能運作的樣子。}
{每個情境包含：前提條件 → 操作步驟 → 預期結果}

### 情境 1: {Happy Path}
- 前提：...
- 操作：...
- 預期：...

### 情境 2: {邊界情境}
- 前提：...
- 操作：...
- 預期：...

### 情境 3: {錯誤處理}
- 前提：...
- 操作：...
- 預期：...

## 影響範圍

### 前端
| 檔案 | 變更類型 | 說明 |
|------|---------|------|
| `js/src/...` | 新增/修改 | ... |

### 後端
| 檔案 | 變更類型 | 說明 |
|------|---------|------|
| `inc/classes/...` | 新增/修改 | ... |

### 資料庫
{是否需要新增欄位/表？如果不需要就寫「無影響」}

## 注意事項

{列出開發時容易踩的坑、需要特別注意的地方。例如：}
{- 與某個現有功能的相容性}
{- 效能考量}
{- 安全性（權限檢查、輸入驗證）}
{- 第三方服務的限制}

## 驗收標準

{可勾選的 checkbox 清單，每一條對應一個可驗證的行為}
- [ ] {具體的驗證項目 1}
- [ ] {具體的驗證項目 2}
- [ ] {具體的驗證項目 3}
```

### 撰寫原則

1. **背景資訊**：簡潔但充分，讓不熟悉這塊功能的開發者也能理解。引用相關的 specs 檔案路徑讓讀者可以深入查閱。

2. **任務目標**：用條列式，每條一個動作。避免模糊的描述如「優化 XX」，要說清楚「把 XX 從 A 改成 B」。

3. **情境案例**：至少 3 個情境 — happy path、邊界情境、錯誤處理。用具體的數值和名稱，不要用抽象的「某個用戶」「某個課程」。參考現有 feature files 中的 Background 資料作為範例數據。

4. **影響範圍**：根據對專案目錄結構的理解，列出可能需要修改的檔案。不確定的標註「待確認」。如果無法判斷就省略此段。

5. **注意事項**：從 Step 3 的發想中提取出開發者最需要注意的 3-5 點。

6. **驗收標準**：每條對應情境案例中的一個預期結果，用 checkbox 格式方便追蹤。

---

## Step 5: 輸出

直接將完整的 markdown 輸出到終端機上。不要寫入檔案，讓用戶自行複製到 GitHub Issue。

輸出前加一行分隔：
```
---
以下是生成的需求文件，可直接複製到 GitHub Issue：
---
```

輸出後提示：
```
---
如需調整，請告訴我哪個段落要修改。
```

---

## 特殊情境處理

### 用戶的 idea 太模糊
如果用戶只說了一句「我想加個 XX 功能」且無法從 specs 中推斷細節，生成需求時：
- 在不確定的地方用 `⚠️ 待確認：...` 標記
- 列出你做了哪些假設，放在需求最前面
- 仍然生成完整結構，讓用戶有東西可以修改

### 用戶的需求與現有功能衝突
在背景資訊中明確指出衝突點，並在注意事項中說明需要如何處理。

### 用戶的需求跨多個模組
如果需求太大，建議拆分成多個 issue，並說明拆分邏輯和依賴順序。
