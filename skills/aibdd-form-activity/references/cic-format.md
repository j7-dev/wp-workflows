# 便條紙格式（CiC — Clarify-in-Context）

## 唯一格式

```
CiC(<CATEGORY>): <詳細的澄清問題，包含上下文、可能選項及影響分析>
```

各視圖的註解前綴：

| 視圖 | 語法 |
|------|------|
| `.activity` | `# CiC(AMB): ...`（行末直接寫，僅 ACTOR / STEP / DECISION / BRANCH / PARALLEL 支援） |
| `.mmd` | `%% CiC(AMB): ...`（Mermaid 註解行） |
| `.feature` | `# CiC(AMB): ...`（Gherkin 註解） |
| `api.yml` | `# CiC(AMB): ...`（YAML 行尾 comment） |
| `erm.dbml` | `// CiC(AMB): ...`（DBML 行尾 comment） |

**嚴禁使用任何其他格式**，包含但不限於：
- `<!-- ?N[類型] ... -->` ❌（舊格式）
- `（待澄清）` / `(待澄清)` ❌
- `(TODO)` / `# TODO` ❌
- `# 待確認` ❌
- `[待補]` ❌

理由：系統使用 Grep `CiC\(` 統一掃描所有視圖的未解便條紙，非此格式的標記將被忽略。

---

## 5 種分類

| 分類 | 代碼 | 顯示名稱 | 何時標記 |
|------|------|---------|---------|
| Ambiguity | `AMB` | 歧義 | 需求存在多種合理解讀（含分支條件可能不完整） |
| Assumption | `ASM` | 假設 | AI 做了一個未經確認的選擇 |
| Gap | `GAP` | 缺失 | 資訊完全缺失，無法確定該步驟的詳細行為 |
| Conflict | `CON` | 矛盾 | 同一流程中前後文衝突、需求之間存在矛盾 |
| Boundary | `BDY` | 邊界 | 範圍邊界不明確（含存在其他可行路徑但暫時未建模） |

**展示規則**：展示給使用者時，一律使用**顯示名稱**（繁體中文），不使用英文代碼。檔案內格式仍維持 `CiC(AMB)` 不變。

**原則**：寧可多標記、少遺漏。不確定就標記，確定了就刪除。

---

## 品質標準

每張便條紙必須「自足」—— 讀者不需查閱其他檔案，就能理解為什麼不確定、有哪些選項、各自的影響。

**必須包含：**
1. **背景**：為什麼這裡有疑問（上下文）
2. **選項**：至少 2 個可能解法
3. **影響**：每個選項對流程結構（STEP 數、DECISION 分支數、綁定檔案數、API schema、DB schema）的影響

**範例對比：**

❌ 低品質（只有結論）：
```
CiC(ASM): 假設僅支援信用卡付款
```

✅ 高品質（自足、含選項與影響）：
```
CiC(ASM): 付款方式假設為信用卡，但需確認支援範圍。
選項 A: 僅信用卡 → DECISION:3a 只有「付款成功/失敗」2 條分支，綁定 1 個 .feature
選項 B: 信用卡 + LINE Pay → DECISION:3a 需增加「選擇付款方式」分支，多 1 個 .feature
選項 C: 含貨到付款 → 還需額外 STEP 處理物流通知事件，影響後續整個流程結構
影響：此決策決定 DECISION:3a 的分支數量與對應 .feature 的數量。
```

**標記原則：寧多勿少、不計 token。每一個不確定就標一張。**
