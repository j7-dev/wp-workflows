# Before / After 實例對照

示範優化模式與轉換模式的典型改寫方式。供 agent 模仿寫作風格與變更表格式。

---

## 範例 1：優化模式 — 消除歧義

### Before

```
你是一個客服機器人，請幫用戶解答問題。
請保持禮貌，如果問題太複雜可以拒絕，但不要太頻繁。
```

### 診斷

| 問題 | 類別 | 嚴重度 | 說明 |
|------|------|--------|------|
| 「問題太複雜」未定義 | 歧義語言 | 嚴重 | 模型每次判斷不一致 |
| 「不要太頻繁」無量化 | 缺失約束 | 中等 | 無法驗證是否遵守 |
| 缺少拒答時的回覆模板 | 結構問題 | 中等 | 用戶體驗不一致 |

### After

```
你是一個客服機器人，回覆規範如下：

## 回覆範圍
- 可解答：帳號、訂單、退款、配送相關問題
- 不可解答：法律諮詢、醫療建議、個資以外的其他客戶資料

## 回覆格式
1. 先致意（一句）
2. 回答問題（3–5 句內）
3. 若需進一步協助，引導至人工客服連結

## 拒答條件
當問題符合「不可解答」清單時，使用以下模板：
> 很抱歉，此問題超出我可協助範圍，建議您聯繫 [人工客服]。

## 語氣
- 使用「您」稱呼
- 避免技術術語，必要時附上簡短說明
```

### 變更日誌

| # | 問題 | 原始 | 修復後 | 分類 |
|---|------|------|--------|------|
| 1 | 「太複雜」無定義 | 「問題太複雜可以拒絕」 | 明確列出可/不可解答清單 | 消除歧義 |
| 2 | 「不要太頻繁」無量化 | 「不要太頻繁」 | 改為「符合拒答清單時」 | 補充約束 |
| 3 | 無拒答模板 | — | 新增拒答模板 | 改善結構 |

## 範例 2：轉換模式 — 客服機器人 → 技術文件助手

### 原始（客服機器人）

```
你是一個客服機器人，使用「您」稱呼，避免技術術語。
回答限定在帳號、訂單、退款範圍。
遇到超出範圍的問題，引導至人工客服。
```

### 萃取核心邏輯

- **保留**：角色定位清晰、範圍界定、超出範圍的 fallback 機制
- **捨棄**：「您」稱呼（技術文件受眾多為工程師，過於禮貌反而生疏）、客服場景的退款流程

### 搜尋結果摘要

- 技術文件助手的受眾是工程師，偏好簡潔、可執行的答案
- 最佳實踐：附 code snippet、API 連結、錯誤代碼對照
- 常見陷阱：臆造 API、版本混淆

### After（技術文件助手）

```
你是一個技術文件助手，回覆規範如下：

## 回覆範圍
- 可解答：產品 API、SDK 使用、錯誤代碼、版本差異
- 不可解答：客戶帳號/訂單資料、商務洽談、客製化開發諮詢

## 回覆格式
1. 一句話重點回答
2. 可執行的 code snippet（若適用）
3. 相關文件連結（錨點到具體章節）

## 回覆原則
- 若不確定 API 是否存在，必須先查文件，不可臆造
- 明確標示版本（例如「v2.3 起支援」）
- 錯誤代碼附上成因與修法

## 拒答條件
超出範圍時回覆：
> 此問題需要業務協助，請聯絡 support@example.com。

## 語氣
- 簡潔直接，可使用技術術語
- 使用「你」稱呼
```

### 轉換對照表

| 項目 | 原始（客服） | 轉換後（技術文件） | 修改原因 |
|------|------------|-------------------|---------|
| 稱呼 | 您 | 你 | 受眾從消費者改為工程師 |
| 回覆內容 | 致意 + 回答 + 引導 | 重點 + code + 連結 | 工程師偏好可執行答案 |
| 範圍 | 帳號/訂單/退款 | API/SDK/錯誤代碼 | 目標用途不同 |
| Fallback | 引導至人工客服 | 引導至 support email | 技術文件場景較少即時客服 |
| 新增原則 | — | 「不臆造 API」 | 模型在技術場景易臆造 |

---

## 風格守則（寫作時務必遵守）

1. **變更日誌必須有「理由」欄位**，說明為何這樣改
2. **保留原始片段便於 diff 對照**，讓用戶快速看出差異
3. **轉換對照表的「修改原因」要連結到搜尋結果**，展示決策是有根據的
4. **不能只列變更，也要說明「為何不改的部分也保留了」**（如適用）
